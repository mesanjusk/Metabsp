# Authentication guide

This platform has **two** separate authentication mechanisms for two
different audiences — don't mix them up.

## 1. JWT Bearer auth (the platform's own frontend, and any first-party integration)

Used by every `/api/whatsapp/*`, `/api/billing/*`, and `/api/bulk/*` route.

```
Authorization: Bearer <jwt>
```

- Obtained by logging in (`POST /api/bulk/auth/login` or the equivalent
  Cloud-product login route) with a mobile number + password.
- Verified on **every request** against the database (`bulk/middleware/auth.js`'s
  `protect`) — not just a claims-only check, so deactivating a user
  (`isActive: false`) immediately invalidates all their sessions on their
  next request, without needing a token blocklist.
- Expires after 30 days (`JWT_EXPIRES_IN` in `backend/.env.example`) — no
  refresh-token endpoint exists; the client re-authenticates (logs in
  again) once expired.
- Never send this token to a third party. It's for this platform's own
  frontend/first-party clients only.

## 2. API key auth (the External API, `/api/v1/*`, for third-party integrations)

Used by `/api/v1/baileys/*` — sending messages programmatically from a
customer's own external system, without a login session.

```
X-Api-Key: mbsp_<key>
```
or
```
GET /api/v1/baileys/status?apiKey=mbsp_<key>
```

- Created from inside the platform (an authenticated user generates one
  for themselves — see `createApiKey`/`listApiKeys`/`revokeApiKey` in
  `backend/src/controllers/whatsappController.js` and the equivalent
  `/api-keys` UI panel).
- Not scoped to an expiry — revoke it explicitly
  (`DELETE /api/v1/api-keys/:id` or the UI's revoke action) if a key is
  compromised, rather than waiting for it to expire.
- Rate-limited: 60 requests/minute on most `/api/v1/baileys/*` routes
  (see `createRateLimiter` usage in `backend/src/routes/externalApi.js`).

## There is no third-party OAuth authorization flow

If you were expecting a "Sign in with <this platform>" OAuth2
authorization-code flow for external apps to request access on a
customer's behalf — **this platform doesn't have one**. The two
mechanisms above (JWT for first-party, API keys for programmatic
third-party access) are the entire authentication surface today. If your
integration needs delegated, revocable, scoped access without ever
handling a customer's password, an API key generated by that customer
inside the platform is the closest equivalent currently available.

## Choosing which one to use

| Building | Use |
|---|---|
| The platform's own web frontend | JWT (already wired up in `frontend/src/apiClient.js`) |
| A customer's external system sending messages via this platform | API key (`/api/v1`) |
| A server-to-server integration you also control end-to-end | Either works — API key avoids needing to manage token refresh |
